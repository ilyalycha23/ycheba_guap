Лабораторная работа №7 — содержание отчёта (методичка.pdf, гл. 7, п. 7.4, вариант 12)

1. Цель
   Изучение механизма триггеров в PostgreSQL: моменты BEFORE / AFTER, события INSERT / UPDATE / DELETE, построчная обработка (FOR EACH ROW), использование записей OLD / NEW.

2. Ход работы (текстом, по шагам)
   Подготовка: **lab1**, **lab2**; при повторе — **09_drop_lr7_objects.sql**.  
   Выполнить **01_aux_audit_table.sql**, затем **02–05** (создание функций и триггеров), затем **06_demo_triggers.sql**; в отчёте последовательно описать, что делали и что наблюдали (сообщения об ошибках, строки аудита).  
   Развёрнутый текст — в **otchet_lr7.docx** (раздел 2).

3. Условия и реализованные триггеры
   Кратко описать назначение каждого объекта (по файлам lab7/02–05):
   — проверка даты приёма (таблица прием);
   — журналирование изменений сотрудника (таблица lr7_employee_audit);
   — лимит числа детей на сотрудника (таблица дети);
   — запрет удаления подразделения при наличии дочерних (таблица подразделение).

4. Тексты и схема срабатывания
   Листинги CREATE FUNCTION / CREATE TRIGGER из lab7/01–05; для каждого — когда срабатывает (какое DML), какие поля читаются из OLD/NEW.

5. Результаты выполнения
   Вывод psql или скриншоты:
   — неуспешная вставка в прием с датой > CURRENT_DATE (сообщение lr7);
   — успешный UPDATE сотрудника и строки в lr7_employee_audit (как в 06_demo_triggers.sql);
   — при необходимости демонстрация DELETE подразделения с дочерними (ошибка lr7).

6. Вывод
   Сравнение с явными ограничениями CHECK / FK: когда триггер оправдан, какие минусы (производительность, отладка, порядок срабатывания).
