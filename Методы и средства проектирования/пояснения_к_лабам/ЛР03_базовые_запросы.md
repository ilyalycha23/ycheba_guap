## `01_queries_abc.sql` — разбор

### Запрос а) (строки 1–13)

**Цель:** сотрудники, числящиеся в заданном подразделении (по **наименованию**).

| Фрагмент | Смысл |
|----------|--------|
| `FROM сотрудник с, прием пр, подразделение пд, должность д` | Старый стиль: **декартово** перечисление таблиц, связь **только** через `WHERE` — важно, чтобы условия дали **эквивалент INNER JOIN** (здесь так и есть). |
| `с.ид = пр.ид_сотрудника` | Привязка приёма к сотруднику. |
| `пр.ид_подразделения = пд.ид` | К какому подразделению относится приём. |
| `пр.ид_должности = д.ид` | Какая должность в приёме (для вывода названия). |
| `пд.наименование = 'Отдел главного конструктора'` | **Фикс** подразделения по тексту; в учебнике вместо этого часто вводят **параметр** (в psql: переменная или подстановка). |
| `ORDER BY с.фио` | Сортировка по ФИО. |

*Если забыть условие связи* — получите декартово произведение: **катастрофа по числу строк**.

### Запрос б) (строки 15–24)

**Цель:** **прямые дети** заданного родителя: все подразделения, у которых `ид_родительское` = ид **«Промтех»** (ищется по `род.наименование`).

- Два раза `подразделение` — `доч` (потомок) и `род` (родитель).
- `доч.ид_родительское = род.ид` — иерархическая связь.
- `род.наименование = 'ООО «Промтех»'` — выбор «корневой» компании.

### Запрос в) (строки 26–34)

**Цель:** сотрудники, у которых есть **хотя бы одна** пара детей **разного пола**.

- `FROM сотрудник с, дети д1, дети д2` — **два** псевдонима `дети` для **двух** строк таблицы (два «ребёнка»).
- `с.ид = д1.ид_сотрудника AND с.ид = д2.ид_сотрудника` — оба ребёнка **этого** сотрудника.
- `д1.пол <> д2.пол` — разные полы.
- `SELECT DISTINCT` — иначе пары (ребёнок A, B) и (B, A) могли бы дать дубли, если схема не аккуратна; здесь `DISTINCT` снимает лишнее.

*Альтернатива* — `EXISTS` (ЛР5), но в ЛР3 вас просили обойтись **без** подзапросов, поэтому — двойной джойн.

## `02_demo_directives.sql` — по блокам

| Блок (стр. примерно) | Что демонстрирует |
|---------------------|-------------------|
| 1–6 | `DISTINCT` + `AS` + `ORDER BY` — уникальные **категории** должностей. |
| 8–14 | `ORDER BY дата_рождения DESC` — **самые молодые** сначала; вторичная сортировка по фио. |
| 16–25 | `NOT IN (2, 4)` — сотрудники **не** в подразделениях 2 и 4 (текущий приём). |
| 27–35 | `BETWEEN дата1 AND дата2` — диапазон **включительно** по дате рождения. |
| 37–43 | `IS NULL` — корневые подразделения (`ид_родительское` пусто). |
| 45–53 | `IS NOT NULL` — только «не корень». |
| 55–62 | `LIKE 'К%'` — фамилия/строка с началом **К** (кодировка/регистр в PostgreSQL с **ILIKE** проще, но в задании — `LIKE`). |
| 64–71 | `NOT LIKE 'С%'` — **не** начинается на С. |

**Важно про `NOT IN`:** если список содержит **NULL**, семантика `NOT IN` может дать **пусто**; в демо только `(2, 4)` — безопасно.
