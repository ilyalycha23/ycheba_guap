# Лабораторная 7 — триггеры (PostgreSQL)

## Задача

Понять **триггер** как подпрограмму, привязанную к событию таблицы:

- **Когда:** `BEFORE` / `AFTER`, `INSERT` / `UPDATE` / `DELETE`.
- **Гранулярность:** `FOR EACH ROW` — на каждую изменённую строку (в лаборатории — он).
- **Данные:** в функции доступны **`NEW`** (новая версия строки) и **`OLD`** (старая; для DELETE — только OLD, для INSERT — только NEW).

В PostgreSQL функция триггера объявляется как **`RETURNS trigger`** и вызывается из **`CREATE TRIGGER ... EXECUTE PROCEDURE имя()`** (в старых версиях формулировка `PROCEDURE`; суть — **одна** функция без параметров в сигнатуре, контекст дают `NEW`/`OLD`/`TG_OP`).

Префикс **`lr7_`**.

## `01_aux_audit_table.sql`

- **Обычная (не TEMP) таблица** `lr7_employee_audit` — журнал, куда `AFTER`-триггер на `сотрудник` **вставит** копии полей.  
- `bigserial` — автоинкремент для `id` журнала.  
- `clock_timestamp()` по умолчанию — **реальное** время (не `transaction_timestamp()`, если важна разница внутри длинной транзакции — для лабы не критично).

## `02_trg_priyom_date_check.sql` — по строкам

| Стр. | Пояснение |
|------|-----------|
| 3 | `CREATE OR REPLACE FUNCTION` — функция, но специфика **trigger**. |
| 4 | `RETURNS trigger` — обязательный тип. |
| 8 | `IF NEW.дата_приема > CURRENT_DATE` — сравнение **вставляемой/обновляемой** даты с **сегодня**. `CURRENT_DATE` в PostgreSQL = дата по часовому поясу сессии. |
| 9–10 | `RAISE EXCEPTION` — отмена **всей** вставки/обновления строки; `ERRCODE` **23514** — «check_violation» (осмысленно для логов). |
| 12 | `RETURN NEW` — в `BEFORE` **обязан** вернуть строку, которая **пойдёт** в таблицу; можно было бы подменить поля `NEW.*`. |
| 16–18 | `DROP TRIGGER IF EXISTS` — пересоздание без дубля. `ON прием` — **имя** таблицы. |
| 18–21 | `BEFORE INSERT OR UPDATE OF дата_приема` — при **UPDATE** триггер срабатывает, только если **менялся** перечисленный столбец. |
| 21 | `FOR EACH ROW` — **не** на весь `INSERT` из `SELECT` как один раз, а **на каждую** вставляемую строку. |
| 21 | `EXECUTE PROCEDURE lr7_fn_priyom_date_check()` — вызов функции **без** аргументов (контекст внутри). |

**Зачем BEFORE:** нельзя вставить «плохую» дату; тихо не пишем и не делаем `AFTER` с откатом.

## `03_trg_employee_audit.sql`

| Фрагмент | Пояснение |
|----------|-----------|
| `TG_OP` | Встроенная переменная: `'INSERT'`, `'UPDATE'`, `'DELETE'`. |
| `IF TG_OP = 'DELETE' THEN` | В журнал пишем **старые** `OLD.фио` / дата. `RETURN OLD` в `AFTER` для DELETE — по доке возвращаемое значение для строкового триггера игнорируется, но **по стилю** пишут. |
| `INSERT` / `UPDATE` | `NEW.ид` **после** вставки уже совпадает с тем, что в таблице. |
| `AFTER` | Срабатывает **после** успешной записи основной строки; если основная вставка откатится — и журнальная вставка откатится (в **той же** транзакции). |

## `04_trg_deti_limit.sql`

- **Перед** `INSERT` в `дети`: `SELECT COUNT(*) ... WHERE ид_сотрудника = NEW.ид_сотрудника` — **уже** сколько детей (новая строка **ещё не** в таблице).
- `v_cnt >= 12` — **13-го** нельзя (на учебных данных сотрудник 6 с 6 детьми — ок).
- `RETURN NEW` в `BEFORE INSERT` — разрешает вставку.

---

## `05_trg_subdept_delete_guard.sql`

- **Перед** `DELETE` из `подразделение`: `OLD.ид` — кого удаляют.
- `SELECT ... WHERE ид_родительское = OLD.ид LIMIT 1` — достаточно **одного** ребёнка, чтобы **запретить**.
- `RETURN OLD` в `BEFORE DELETE` — подтвердить удаление; если RAISE — удаления нет.

**Связь с FK:** СУБД **тоже** не даст удалить родителя с дочерними по FK, но **сообщение** пользователю может быть нечитабельным; триггер даёт **пояснение** раньше (или **вместо** сырой ошибки, в зависимости от порядка проверок — в лабе это демо).

## `06_demo_triggers.sql` — смысловые куски

- Блок `DO` с `EXCEPTION` — вставка «плохой» даты **ловится**, `NOTICE` с текстом.
- `BEGIN; UPDATE сотрудник; SELECT из lr7_employee_audit; ROLLBACK;` — увидеть строку **в рамках** незавершённой транзакции, затем **откат** убирает и сотрудника, и строку **аудита** (тот же txn).

## `09_drop_lr7_objects.sql`

- Сначала **`DROP TRIGGER ... ON имя_таблицы`**, потом **`DROP FUNCTION`** — иначе функция всё ещё «занята».
- `DROP TABLE lr7_employee_audit` **после** удаления триггера, который писал в неё (на `сотрудник`).

---

## Итог

- **BEFORE** — валидация, подмена `NEW`, запрет.  
- **AFTER** — лог «что сделалось» без блокировки самой DML, если лог **не** должен влиять на допустимость.  
- **NULL / сравнения** в триггерах — те же правила, что в обычном SQL.
